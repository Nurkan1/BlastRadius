BlastRadius — Help & MCP Integration

Connect any MCP-capable agent to query your live iteration state.

BlastRadius exposes a read-only Model Context Protocol server at http://localhost:7842/mcp. Pick your agent below and run the matching command to register the server. Every command is idempotent and creates a timestamped backup before overwriting any existing configuration.

Claude Code (CLI)

Recommended when the claude binary is on your PATH.

claude mcp add --transport http blastradius http://localhost:7842/mcp
claude mcp list
claude mcp get blastradius

Claude Desktop

Claude Desktop only accepts the stdio transport. Use the installer to register the bundled shim under a server name that bypasses Claude Desktop's persistent rejection list (see the Troubleshooting tab): 

.\scripts\install-hook.ps1 -ProjectPath . -Agent claude -RegisterDesktop

After the command finishes, fully quit Claude Desktop from the system tray and reopen it. The new server appears as blastradius-observability in your connected MCP list.

Antigravity 2.0

.\scripts\install-hook.ps1 -ProjectPath . -Agent antigravity -RegisterMcp

Then run /reload in the Antigravity session — Antigravity does not hot-reload MCP configuration.

Custom Anthropic SDK client

import { Client } from '@modelcontextprotocol/sdk/client/index.js'
import { StreamableHTTPClientTransport } from '@modelcontextprotocol/sdk/client/streamableHttp.js'

const transport = new StreamableHTTPClientTransport(
  new URL('http://localhost:7842/mcp'),
)
const client = new Client({ name: 'my-app', version: '1.0.0' })
await client.connect(transport)

const summary = await client.callTool({
  name: 'get_iteration_summary',
  arguments: {},
})
console.log(summary.structuredContent)

The MCP server exposes ten read-only tools, one mutating tool guarded by requiresConsent, and nine resources (plus one templated). Every payload follows the same NO-DATA contract: tools never throw on absence — instead they return a structured object whose value fields are null and whose reason is a stable machine-readable string.

Tools — observability (Phase 1, rc7)

NameReturns
get_iteration_summary Current iteration metrics and per-file activity attribution.
summarize_progress Per-file Edit / Write / Read aggregation between since and until (both optional).
list_recent_iterations Iteration windows inferred from activity-gap boundaries (default 3 min).
list_days_with_activity Every day with a session-*.jsonl on disk, sorted desc, capped at 30. (rc7+)
get_file_diff Validated git diff of one repo-relative file. Path validation is shared with /api/diff.

Tools — Knowledge Graph (rc8+)

NameReturns
get_codebase_graph Knowledge Graph nodes + edges with limit / kinds / minFanIn / withSummaryOnly filters. Default 200, hard ceiling 1000.
get_nearest_neighbors BFS consumers / dependencies for a node, depth ≤ 10. Path validation reused from /api/diff.
describe_node Full node detail (kind, size, fanIn / fanOut, summary, tags) plus a 7-day touch-event cross-walk.
find_nodes Text search ranked path startsWith > tag exact > summary contains > path contains.
set_node_summary write Persist a summary (≤ 2000 chars) + tags (≤ 20 × 32) to ~/.blastradius/knowledge.json. Carries annotations.requiresConsent: true so MCP clients can prompt before firing it.

NO-DATA reason codes

When a tool or resource cannot produce a meaningful answer, the response carries one of these stable reason strings:

  • no_active_repo — preferences have no current repo set.
  • needs_setup — dashboard is still in the first-run wizard.
  • no_active_iteration — iteration marker has never been closed.
  • no_events_in_window — the time window contains zero events.
  • no_events_recorded — the event store is empty today.
  • unknown_window — the caller requested an unsupported window.
  • escapes_root / path_traversal / nul_byte / absolute_path — path validation refused the input.
  • invalid_ref — the git ref did not match the allowed character set.
  • graph_not_ready — KnowledgeGraph snapshot has not been built yet. (rc8+)
  • unknown_node — the path is valid but absent from the current graph snapshot. (rc8+)
  • no_matches — find_nodes returned zero results. (rc8+)
  • cycles_none — the graph is a clean DAG. (rc8+)
  • orphans_none — every non-entry-point file has at least one consumer. (rc8+)
  • summary_too_long / too_many_tags / tag_too_long / tag_invalid_type — set_node_summary cap defense. (rc8+)

Drop any of these prompts directly into a session with an MCP-connected agent. The model will pick the right tool or resource on its own — you do not have to mention them by name.

Bootstrap prompt (recommended at session start)

You have access to BlastRadius, a read-only observability MCP server
that tracks the files Claude Code, Antigravity, and other agents touch
in this repository. Use it whenever you need to ground a question in
real activity — current iteration progress, hottest files today, recent
diffs, or which agent edited what. Prefer BlastRadius over guesswork
when the question is about *what just happened* in the repo.

Iteration summary

Summarize my current BlastRadius iteration. List the edited files,
the read files, and any files affected by import propagation.

Today's heat map

Read blastradius://heat/session and tell me which files are red and
why. Identify any unexpected hot spots that suggest scope creep.

Progress digest since a checkpoint

Use BlastRadius summarize_progress with since=2026-01-01T08:00:00Z
to give me an end-of-morning digest: top 5 files by edit count, total
reads vs writes, and which agent touched each one.

Iteration history

List my last 5 BlastRadius iterations and characterize each one in
one sentence based on the files touched and the duration.

Inline diff review

For every file currently marked red in BlastRadius, fetch its diff via
get_file_diff and tell me whether the change is safe to commit.

Cross-check with no-data contract

Call get_iteration_summary. If the response has reason
"no_events_in_window", do not invent activity — say "no events in the
current iteration" and suggest closing the iteration or waiting.

End-of-day digest (rc7+)

Use list_days_with_activity to find the most recent day with logs.
Then call summarize_progress with since=<that day>T00:00:00Z and
until=<that day>T23:59:59Z. Give me a one-paragraph digest of the
top 5 files by edit count, total reads/writes, and which agent did
most of the work.

Weekly review (rc7+)

List my last 7 days of activity via list_days_with_activity, then
for each non-empty day call summarize_progress with since/until bound
to that single day. Build a weekly table: date, total events, top
file, dominant agent. Highlight the day with the highest blast
radius.

Impact analysis before a refactor (rc8+)

I'm about to refactor src/server/heatEngine.js. Use BlastRadius
get_nearest_neighbors with direction="consumers" and depth=3 to
list every file that imports it transitively. Then describe_node
each of those files so I know what they expect from heatEngine
before I touch its public API.

Dead-code review (rc8+)

Read blastradius://graph/orphans. For each orphan path, run
describe_node and tell me whether it looks like an entry point
the allowlist missed, a test fixture, or genuine dead code.
Group your conclusions and propose deletions only for the
"genuine dead code" pile.

Cycle detection + remediation (rc8+)

Read blastradius://graph/cycles. If reason is "cycles_none",
say so and stop. Otherwise, for each cycle, identify the
weakest edge (the file with the lowest fanIn) and propose
a refactor that breaks the cycle without changing public
behavior of the other nodes.

Persist what you've learned (rc8+, mutation — needs consent)

After understanding src/server/heatEngine.js, call set_node_summary
to persist a one-paragraph description of what it does plus the
tags ["core", "pure"]. Future sessions will see your summary via
describe_node so we don't have to re-explore the file from scratch.

Four real-world quirks discovered during empirical integration with Claude Code, Claude Desktop, and Antigravity 2.0.

Claude Desktop rejects type: http

claude_desktop_config.json only accepts the stdio shape — { "command": "node", "args": [...] }. HTTP entries are silently dropped with the warning "Skipped invalid MCP server config entries" in the main log. Use the bundled bin/blastradius-mcp.cjs shim, which proxies stdio JSON-RPC to the dashboard's HTTP MCP transport.

Claude Desktop holds a persistent rejection list by name

Once Claude Desktop rejects an entry under a given server name, it keeps deleting that name from the config on every subsequent read, even after the shape is corrected. The workaround is to register under a different name; the installer's -RegisterDesktop uses blastradius-observability for exactly this reason.

.mjs entries are filtered out

Claude Desktop's config validator rejects any args entry that points at a .mjs file. Production traffic therefore goes through the .cjs wrapper, which spawns the .mjs implementation as a child process and inherits its stdio.

"BlastRadius dashboard not running" errors

If the agent reports upstream unreachable at http://localhost:7842/mcp, the dashboard is not running on the default port. Either open the dashboard, or set BLASTRADIUS_PORT and pass -McpUrl http://localhost:<port>/mcp to install-hook.ps1 when registering.

Recovery from a corrupted config

Every destructive write produces a timestamped backup adjacent to the file (filename.json.bak.YYYYMMDD-HHMMSS). To roll back, copy the most recent backup over the live config and restart the affected agent.

Activate observability in this repo

BlastRadius captures Edit / Read / Write events via a Claude Code PostToolUse hook registered in .claude/settings.json. This action writes a single hook entry pointing at log-touch.js in your BlastRadius install. Existing settings.json content is preserved; if the file already exists a timestamped .bak copy is created next to it.

You'll need to restart any Claude Code session open in this repo for the hook to take effect (Claude Code reads settings.json only at session start).
